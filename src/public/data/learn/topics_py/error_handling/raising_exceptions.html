Raising Exceptions

Handling errors is good. Creating meaningful errors is better. When your code detects something invalid, it should not continue silently — it should raise an exception.

1. Why Raise Exceptions?

So far, we've handled errors that Python raised automatically. But what if you detect something wrong?

Example:

  • A negative age
  • A password that’s too short
  • A withdrawal amount greater than account balance

If your function receives invalid input, it should stop immediately. That’s where raise comes in.

Core Principle
If something violates the rules of your program, fail early and clearly.

2. Basic Syntax of raise

The raise keyword lets you trigger an exception manually. 

age = -5

if age < 0:
    raise ValueError("Age cannot be negative.")

What happens?

  • If age is negative → program stops.
  • Python shows a ValueError with your custom message.

You are enforcing correctness.

3. Raising Exceptions Inside Functions

This is where it becomes useful. 

def withdraw(balance, amount):
    if amount > balance:
        raise ValueError("Insufficient funds.")
    return balance - amount

print(withdraw(100, 150))

Instead of silently returning something incorrect, the function stops execution.

Design Insight
Functions should protect themselves. Never assume the caller passes valid input.

4. Choosing the Right Exception Type

Use built-in exception types when possible.

  • ValueError → correct type, wrong value
  • TypeError → wrong type entirely
  • IndexError → invalid index
  • RuntimeError → general runtime failure

Example: 

def square(number):
    if not isinstance(number, (int, float)):
        raise TypeError("Input must be a number.")
    return number ** 2

Type validation + clear error messaging = professional code.

5. Raising Exceptions After Catching Them

Sometimes you catch an error — then decide it’s still critical. You can re-raise it. 

try:
    value = int("abc")
except ValueError:
    print("Conversion failed.")
    raise

Here:

  • We log or print something.
  • Then re-raise the same exception.

This preserves the original traceback.

6. Defensive Programming Mindset

Raising exceptions is not being dramatic. It is enforcing contracts.

When you write a function, you are making a promise:

  • If valid input → I return a correct result.
  • If invalid input → I fail clearly and immediately.
Professional Reality
Silent failures are worse than crashes. Clear exceptions are easier to debug than incorrect results.

7. Mini Exercise

Write a function set_password(password) that:

  • Raises ValueError if the password is less than 8 characters
  • Raises TypeError if password is not a string
  • Returns "Password set successfully" otherwise

The goal is not complexity. The goal is enforcing rules.
