Custom Exceptions

Built-in exceptions are useful. But as your programs grow, generic errors become vague. Custom exceptions let you describe exactly what went wrong in your own system.

1. Why Create Custom Exceptions?

Imagine building a banking system.

  • Is ValueError descriptive enough for a failed withdrawal?
  • Does RuntimeError clearly explain a failed authentication?

In small programs, built-in exceptions are fine. In larger systems, vague errors become confusing.

Core Idea
Custom exceptions make your code readable, structured, and self-explanatory.

2. Creating a Custom Exception

Custom exceptions are just classes that inherit from Exception. 

class InsufficientFundsError(Exception):
    pass

That’s it. You’ve created a new exception type.

Now you can raise it: 

def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFundsError("Not enough balance.")
    return balance - amount

This is clearer than using ValueError. It communicates business logic.

3. Catching Custom Exceptions

You handle them like any other exception. 

try:
    print(withdraw(100, 150))

except InsufficientFundsError as e:
    print("Transaction failed:", e)

Now your program distinguishes:

  • Logical system errors
  • Unexpected Python runtime errors

This separation is powerful in larger systems.

4. Adding Extra Information

You can customize your exception further by defining an __init__ method. 

class InvalidAgeError(Exception):
    def __init__(self, age):
        self.age = age
        super().__init__(f"Invalid age provided: {age}")

Now: 

def register(age):
    if age < 0:
        raise InvalidAgeError(age)
    return "Registered successfully"

The exception carries structured data. That’s professional-level design.

5. Inheriting from Other Exceptions

You don’t always have to inherit directly from Exception. You can subclass more specific types. 

class DatabaseConnectionError(RuntimeError):
    pass

Now:

  • It behaves like a RuntimeError
  • But has a domain-specific meaning

This is common in frameworks and large applications.

6. When Should You Use Custom Exceptions?

Use them when:

  • Your project is growing beyond simple scripts
  • You have domain-specific rules
  • You want clearer debugging
  • You’re designing APIs or reusable modules

Avoid them when:

  • The program is tiny
  • A built-in exception is already perfectly descriptive
Design Rule
Don’t invent new exceptions just to look advanced. Use them when they improve clarity.

7. Mini Exercise

Create a custom exception called WeakPasswordError.

Then:

  • Raise it if a password has fewer than 8 characters
  • Raise it if the password has no digits
  • Catch it and print a meaningful message

Focus on clarity and structure.

8. Big Picture

Built-in exceptions help you manage Python-level errors. Custom exceptions help you manage system-level rules.

Strategic Insight
As systems scale, error clarity becomes more important than clever code.
