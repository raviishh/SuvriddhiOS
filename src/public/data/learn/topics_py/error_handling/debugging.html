Debugging Techniques

Bugs are not accidents — they are inevitable. Debugging is the disciplined process of locating, understanding, and eliminating them.

1. What Is Debugging?

Debugging is not randomly changing code until it works. That is guessing.

Debugging is:

  • Reproducing the problem
  • Reading the error carefully
  • Forming a hypothesis
  • Testing that hypothesis
  • Fixing the root cause
Core Principle
Never fix what you don’t understand.

2. Read the Traceback — Slowly

When Python throws an error, it gives you a traceback. Most beginners skim it. Professionals study it. 

def divide(a, b):
    return a / b

print(divide(10, 0))

The traceback tells you:

  • Which file
  • Which line number
  • Which function
  • What error occurred

Start at the bottom of the traceback. That’s where the actual exception is described.

3. Strategic Print Debugging

Sometimes you don’t know what values your variables hold. Print them. 

def calculate_total(price, quantity):
    print("Price:", price)
    print("Quantity:", quantity)
    return price * quantity

This helps you verify assumptions.

But be strategic:

  • Print inputs
  • Print intermediate calculations
  • Print before suspicious lines
Warning
If your code has 50 random print statements, you're not debugging — you're panicking.

4. Using assert for Sanity Checks

The assert statement checks assumptions. If the condition is false, it raises an AssertionError. 

def sqrt(number):
    assert number >= 0, "Number must be non-negative"
    return number ** 0.5

If someone passes a negative number, the program stops immediately.

Assertions are useful during development to catch impossible states.

5. Using the Python Debugger (pdb)

Python has a built-in debugger: pdb. It lets you pause execution and inspect variables. 

import pdb

def buggy_function():
    x = 10
    y = 0
    pdb.set_trace()
    return x / y

buggy_function()

When execution reaches set_trace(), you can:

  • Inspect variables
  • Execute code line-by-line
  • Evaluate expressions

Common commands:

  • n → next line
  • c → continue
  • p variable → print variable
  • q → quit debugger

This is structured debugging — not guessing.

6. Rubber Duck Debugging

Explain your code out loud. Line by line.

When you articulate what your program is supposed to do, you often notice where logic breaks.

Why It Works
Bugs hide in assumptions. Speaking forces clarity.

7. Isolate the Problem

If a large program fails:

  • Comment out sections
  • Test smaller pieces independently
  • Reduce to a minimal reproducible example

If the bug disappears in a smaller example, the issue is likely in integration.

Large systems require isolation thinking.

8. Common Debugging Mistakes

  • Changing multiple things at once
  • Ignoring the traceback
  • Suppressing errors instead of fixing them
  • Assuming the problem is somewhere else

Debugging is a discipline. Not an emotional reaction.

9. Mini Exercise

Debug this function:

def average(numbers):
    total = 0
    for i in range(len(numbers)):
        total += i
    return total / len(numbers)

print(average([10, 20, 30]))

It runs — but produces the wrong result.

Use:

  • Print debugging
  • Assertions
  • Reasoning about the loop

Find the logical mistake.

10. Big Picture

You will spend more time debugging than writing new code. That is normal.

Professional Insight
Strong developers don’t avoid bugs. They develop systematic ways to eliminate them.
