try / except / else / finally

Errors will happen. Professional code doesn’t hope they won’t — it plans for them. Python gives us structured tools to respond intelligently instead of crashing.

1. Why We Need Error Handling

By default, when a runtime error occurs, Python stops the program immediately. That’s fine for small scripts — but unacceptable in real systems.

Imagine:

  • A banking app crashing because someone typed a letter instead of a number.
  • A game shutting down because a file wasn’t found.
  • A robot freezing because of a sensor glitch.

Core Idea
Instead of letting the program crash, we intercept the error and decide what to do.

2. The Basic Structure: try and except

The try block contains code that might fail. The except block runs if an error occurs. 

try:
    number = int(input("Enter a number: "))
    print(10 / number)
except ZeroDivisionError:
    print("You cannot divide by zero!")

What happens here?

  • If the user enters 0 → Python raises ZeroDivisionError.
  • The program does not crash.
  • The except block handles it.

Important Rule
Always catch specific exceptions. Avoid using a bare except:.

3. Catching Multiple Errors

Different failures require different responses. 

try:
    number = int(input("Enter a number: "))
    result = 10 / number
    print(result)

except ValueError:
    print("That was not a valid number.")

except ZeroDivisionError:
    print("You cannot divide by zero.")

Notice:

  • ValueError → invalid conversion from string to int
  • ZeroDivisionError → mathematical impossibility

Precise handling makes code predictable.

4. The else Block — When Nothing Goes Wrong

The else block runs only if no exceptions occurred. 

try:
    number = int(input("Enter a number: "))
    result = 10 / number

except ValueError:
    print("Invalid input.")

except ZeroDivisionError:
    print("Cannot divide by zero.")

else:
    print("Result:", result)

Why use else?

  • Keeps error-handling separate from success logic
  • Makes code cleaner
  • Prevents accidental execution if something failed

Design Insight
Put only risky code inside try. Put normal logic in else.

5. The finally Block — Always Runs

The finally block executes no matter what — error or not. 

try:
    file = open("data.txt")
    content = file.read()

except FileNotFoundError:
    print("File not found.")

finally:
    print("Execution finished.")

Even if an error happens, finally runs.

Common uses:

  • Closing files
  • Releasing resources
  • Cleaning up connections

Professional Standard
If your code opens something (file, connection, socket), it must close it. finally guarantees cleanup.

6. The Full Structure

All together: 

try:
    # risky code
    pass

except SpecificError:
    # handle known error
    pass

else:
    # runs if no exception occurred
    pass

finally:
    # always runs
    pass

You do not need all four every time. But understanding them makes you deliberate instead of reactive.

7. What NOT to Do

This is dangerous: 

try:
    risky_operation()
except:
    pass

Why?

  • It hides errors.
  • It makes debugging harder.
  • It can mask serious bugs.

Catching everything without handling it is not defensive programming. It is avoidance.

8. Mini Exercise

Write a program that:

  • Asks the user for two numbers
  • Divides the first by the second
  • Handles invalid input
  • Handles division by zero
  • Prints the result only if successful
  • Always prints "Program ended."

This forces you to use try, except, else, and finally.
