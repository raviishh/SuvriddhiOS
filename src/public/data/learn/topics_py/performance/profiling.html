Profiling in Python (cProfile)

Before you try to make code faster, you must measure it. Profiling helps you see where your program actually spends time.

1. Why guessing is dangerous

Many programmers guess which part of their code is slow. Most of the time, they guess wrong.

Analogy
Imagine your house has a water leak. You do not break random walls. First, you find exactly where the leak is.

Profiling is how we find the “leak” in performance.

2. What is profiling?

Profiling means measuring how long each part of your program takes to run.

Python has a built-in tool called cProfile.

It tells you:

  • Which functions were called
  • How many times they were called
  • How much time they used

3. How to use cProfile

Suppose we have this file called slow_program.py. 

def slow_function():
    total = 0
    for i in range(1000000):
        total += i
    return total

def fast_function():
    return sum(range(1000000))

slow_function()
fast_function()

To profile it, run this in the terminal: 

python -m cProfile slow_program.py

Python will print a report.

4. Understanding the output

The output may look like this (simplified): 

ncalls  tottime  percall  cumtime  percall  function
1       0.200    0.200    0.200    0.200    slow_function
1       0.020    0.020    0.020    0.020    fast_function

Important columns:

  • ncalls — number of times the function ran
  • tottime — time spent inside the function itself
  • cumtime — total time including other functions it called

Here we can see that slow_function takes much more time than fast_function.

Key Idea
Look for functions with large cumtime. That is usually where your program is slow.

5. Profiling inside a script

You can also use cProfile directly inside Python: 

import cProfile

def slow():
    total = 0
    for i in range(1000000):
        total += i
    return total

cProfile.run("slow()")

This runs the function and prints the profiling report.

6. Practice

Try this:

  • Create two functions that solve the same problem in different ways.
  • Profile both versions.
  • Compare the tottime and cumtime.
Friendly Tip
Do not optimize first. Measure first. Then improve only the slow parts.
