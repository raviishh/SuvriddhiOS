Code Refactoring in Python

Refactoring means improving your code without changing what it does. The output stays the same. The behavior stays the same. Only the structure becomes better.

1. What is Refactoring?

Refactoring is cleaning up your code.

You do NOT add new features. You do NOT change results. You only improve the design.

Simple Analogy
Imagine your room is messy. You clean and organize it. The room is the same room — just cleaner.

2. Why Refactor?

Over time, code can become:

  • Too long
  • Hard to read
  • Repeated in many places
  • Difficult to maintain

Refactoring makes code:

  • Cleaner
  • Easier to understand
  • Easier to change later

3. Example: Messy Code

def calculate_total(price, tax):
    total = price + (price * tax)
    print("Total price is:", total)
    return price + (price * tax)

Problems:

  • The calculation is written twice
  • The function both prints and returns
  • Not very clear

4. Refactored Version

def calculate_total(price, tax):
    total = price + (price * tax)
    return total

Now:

  • The calculation is written once
  • The function has one clear job
  • The code is easier to read

If we want to print the result, we can do it outside the function. 

total = calculate_total(100, 0.1)
print("Total price is:", total)

5. Important Rule: Refactor After Testing

You should refactor only when tests exist.

Why?

Safety Net
Tests protect you. After refactoring, run the tests. If they pass, your changes are safe.

Refactoring without tests is risky.

6. Common Refactoring Techniques

  • Rename unclear variables
  • Break long functions into smaller ones
  • Remove repeated code
  • Improve formatting

Example of better variable names:

# Bad
def f(x, y):
    return x * y

# Better
def multiply(number1, number2):
    return number1 * number2

7. Small Exercise

Refactor this function:

def process(numbers):
    result = 0
    for n in numbers:
        result = result + n
    print("Sum is:", result)
    return result

Goals:

  • Improve variable names
  • Remove unnecessary parts
  • Keep the output the same
Final Thought
Good programmers write code. Great programmers improve it.
