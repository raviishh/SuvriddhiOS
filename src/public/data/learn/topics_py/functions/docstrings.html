Docstrings & Documentation in Python

Documentation explains what your code does, how to use it, and why it exists. Docstrings are the primary built-in way to document functions, classes, and modules in Python.

1. What Is a Docstring?

A docstring is a string literal placed immediately after a function, class, or module definition. It describes the purpose and usage of that object. 

def greet(name):
    """Return a greeting message for the given name."""
    return f"Hello, {name}"

Docstrings use triple quotes (""" or ''').

Key Distinction
Comments explain implementation details. Docstrings explain usage and behavior.

2. Accessing Docstrings

Docstrings are stored in the __doc__ attribute. 

print(greet.__doc__)

They are also used by the built-in help() function. 

help(greet)

This makes docstrings part of Python’s introspection system.

3. Writing Effective Docstrings

A good docstring should:

  • Describe what the function does
  • Explain parameters
  • Specify return values
  • Mention exceptions (if relevant)
def add(a, b):
    """
    Add two numbers.

    Parameters:
        a (int or float): First number.
        b (int or float): Second number.

    Returns:
        int or float: The sum of a and b.
    """
    return a + b

4. One-Line vs Multi-Line Docstrings

One-Line Docstring

def square(x):
    """Return the square of x."""
    return x * x

Multi-Line Docstring

def divide(a, b):
    """
    Divide a by b.

    Raises:
        ZeroDivisionError: If b is zero.
    """
    return a / b

Use one-line docstrings for simple functions. Use multi-line format for more complex behavior.

5. Module and Class Docstrings

Modules and classes can also include docstrings. 

"""Utility functions for mathematical operations."""

class Calculator:
    """Perform basic arithmetic operations."""
    pass

These help organize large codebases.

6. Documentation Conventions

Python projects often follow documentation styles such as:

  • Google style
  • NumPy style
  • reStructuredText (Sphinx)

Consistency is more important than the specific format.

Best Practice
Document public functions and APIs. Avoid excessive documentation for trivial internal code.

7. Common Mistakes

  • Writing comments instead of docstrings
  • Describing how instead of what
  • Forgetting to update documentation after code changes
  • Over-documenting trivial functions

8. Why Documentation Matters

Documentation transforms code from a private script into a reusable system. It enables collaboration, maintainability, and scalability.

Well-documented code communicates intent clearly — reducing confusion and future debugging effort.

Practice Strategy
Add proper docstrings to an existing project. Ensure each function clearly states purpose, parameters, and returns.
