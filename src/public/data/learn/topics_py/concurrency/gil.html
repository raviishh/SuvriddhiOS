The GIL — Why Threads Don’t Always Make Things Faster

In this lesson, we explain the GIL in a simple way. You will understand why threads are great for waiting tasks, but not always good for heavy calculations.

1. What is the GIL?

GIL means Global Interpreter Lock.

It is a rule inside Python that says:

Only one thread can run Python code at a time.

Even if you create many threads, only one thread runs at any moment.

Simple Analogy
Imagine a classroom with one whiteboard and many students. Only one student can write on the board at a time. That rule is like the GIL.

2. Why does the GIL exist?

Python uses the GIL to keep memory safe and simple.

Without it, Python would be much more complex. The GIL prevents many memory errors.

So the GIL is not a mistake. It makes Python easier and safer.

3. CPU-bound vs I/O-bound tasks

The GIL affects different types of tasks in different ways.

CPU-bound task:

  • Heavy calculations
  • Large loops
  • Math operations

I/O-bound task:

  • Waiting for internet
  • Reading files
  • Waiting for user input
Important Idea
The GIL blocks CPU-bound threads from running at the same time. But it does NOT block waiting tasks in the same way.

4. Example: CPU-heavy threads

import threading

def heavy_task():
    total = 0
    for i in range(100000000):
        total += i
    print("Task finished")

t1 = threading.Thread(target=heavy_task)
t2 = threading.Thread(target=heavy_task)

t1.start()
t2.start()

t1.join()
t2.join()

print("Done")

You might expect this to run twice as fast. But usually, it does NOT.

Why? Because the GIL allows only one thread to run Python code at a time.

5. When does the GIL matter?

The GIL matters when:

  • You use threads
  • Your program is CPU-heavy

The GIL does NOT matter much when:

  • Your program is waiting (I/O tasks)
Simple Rule
CPU work → use multiprocessing. Waiting work → threads are fine.

6. Final Summary

Let’s summarize:

  • The GIL allows only one thread to run Python code at a time.
  • Threads do not speed up heavy calculations.
  • Threads are still useful for waiting tasks.
  • Use multiprocessing for real parallel CPU work.
Don’t panic
The GIL sounds scary, but it is simple: it only matters for heavy CPU tasks with threads.
