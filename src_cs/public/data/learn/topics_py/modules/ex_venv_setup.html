Set Up a Project with venv & requirements.txt

Practice the reasoning and file formats behind virtual environments and dependency pinning, without needing a real terminal.

Part 1 — Writing a requirements.txt

requirements.txt lists a project's dependencies, one per line, optionally pinned to exact versions with ==.

requirements = """requests==2.31.0
flask==3.0.0
"""
print(requirements.strip())
Output:
requests==2.31.0
flask==3.0.0

What to do

  • Given a project that uses requests version 2.31.0, pandas version 2.1.4, and pytest version 7.4.3 (dev-only), build the exact text content of a requirements.txt file with each package pinned using ==, one per line, in alphabetical order by package name.
  • Print the resulting string.

Watch out for

  • Do not add a trailing blank line to the printed output — the checker compares exact text.
  • Match the version strings exactly as given, including all digits.
Hints (4)
  • Build a list of tuples [('requests', '2.31.0'), ('pandas', '2.1.4'), ('pytest', '7.4.3')], sort it by name, then format each as f'{name}=={version}'.
  • '\n'.join(lines) produces the final file content, one dependency per line.
  • Alphabetical order: pandas, pytest, requests.
  • In real projects, dev-only dependencies like pytest are sometimes split into a separate requirements-dev.txt, but this exercise keeps them together for simplicity.

Part 2 — Diagnosing a venv Problem

A very common beginner bug: installing a package with pip while the virtual environment isn't activated, so the package installs globally rather than into the project's isolated environment.

# Symptom: 'pip install requests' succeeds, but 'import requests' still
# raises ModuleNotFoundError inside the project's venv.
# Likely cause: the venv was never activated before running pip.

What to do

  • Write a function diagnose(venv_active, package_installed_globally, import_error) that returns the most likely cause as a short string, given these three booleans.
  • If venv_active is False and package_installed_globally is True and import_error is True: return 'venv not activated — package installed globally instead'.
  • If venv_active is True and import_error is True and package_installed_globally is False: return 'package not installed in this venv — run pip install again'.
  • Otherwise return 'no obvious issue detected'.
  • Test diagnose(False, True, True), diagnose(True, False, True), and diagnose(True, True, False), printing each result.

Watch out for

  • Copy the return strings exactly — the checker is exact-match, including the — character (em dash, not a hyphen).
  • Test all three calls in the order given so the printed lines match the expected output order.
Hints (4)
  • This is a straightforward if/elif/else chain over the three boolean inputs.
  • Order your conditions to check the most specific combinations first.
  • The exact returned strings must match char-for-char, including the em dash and punctuation.
  • diagnose(True, True, False) means the venv is active, the package IS installed, and there's no import error — hence 'no obvious issue detected'.

Stretch — Dependency Conflict Resolver stretch

When two packages require incompatible version ranges of a shared dependency, pip (or a human) must find a version satisfying both, or report that none exists.

# package_a needs shared>=2.0,<3.0
# package_b needs shared>=2.5,<4.0
# Overlap: [2.5, 3.0) — any version in that range satisfies both.

What to do

  • Write a function resolve(range_a, range_b) where each range is a tuple (min_inclusive, max_exclusive), and it returns the overlapping range as a tuple, or None if there is no overlap.
  • resolve((2.0, 3.0), (2.5, 4.0)) → (2.5, 3.0)
  • resolve((1.0, 2.0), (2.0, 3.0)) → None (no overlap since the first range's max is exclusive and equals the second's min)
  • resolve((1.0, 5.0), (2.0, 3.0)) → (2.0, 3.0)
  • Print all three results.

Watch out for

  • Since the max in each range is exclusive, an overlap_min equal to overlap_max means zero width — treat that as no overlap (return None), not a valid single-point range.
  • Print None (the Python object), not the string 'None', for the second case — print(None) naturally shows 'None' as text, which is fine.
Hints (4)
  • overlap_min = max(range_a[0], range_b[0]); overlap_max = min(range_a[1], range_b[1])
  • If overlap_min >= overlap_max, there is no valid overlapping range — return None.
  • Otherwise return (overlap_min, overlap_max) as a tuple.
  • This models exactly what pip's dependency resolver does at a conceptual level, just simplified to single ranges.
