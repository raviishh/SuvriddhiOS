Refactor a Script into a Package

Simulate splitting a single flat script into logically separated modules using in-memory module objects.

Part 1 — Separating Concerns

Splitting code into modules means grouping related functions together and importing what you need. A module is just a namespace — you can build one dynamically to demonstrate the concept.

import types
mathutils = types.ModuleType("mathutils")
exec("def square(n): return n * n", mathutils.__dict__)
print(mathutils.square(4))
Output:
16

What to do

  • Create a module object named 'shapes' containing two functions: area_circle(r) returning 3.14159 * r ** 2, and area_square(s) returning s ** 2.
  • Create a second module object named 'formatting' containing one function: money(x) returning f'${x:.2f}'.
  • Using both modules together, print formatting.money(shapes.area_circle(3)) and formatting.money(shapes.area_square(4)).

Watch out for

  • In a real project you would create actual .py files and use 'from shapes import area_circle' — this exercise simulates that boundary in a single file.
  • Keep the rounding to 2 decimal places via the :.2f format spec, not manual round().
Hints (4)
  • Use types.ModuleType('name') then exec(code_string, module.__dict__) to define functions inside it, simulating a real module file.
  • 3.14159 * 3 ** 2 = 28.27311 which rounds to 28.27 under :.2f formatting.
  • This mirrors how, in a real project, shapes.py and formatting.py would each be separate files imported with 'import shapes'.
  • Access functions via moduleobj.function_name(...), exactly like a real imported module.

Part 2 — Avoiding Circular Imports

Circular imports happen when module A imports module B and module B imports module A. The fix is usually to move the shared logic into a third module, or to import inside a function rather than at the top of the file.

# BAD (circular):
# a.py: from b import helper
# b.py: from a import CONFIG
#
# GOOD: move CONFIG to a shared config.py that both import from

What to do

  • Explain, by writing it as a printed multi-line string, why the following would cause a circular import: module 'orders' needs a function from 'inventory', and 'inventory' needs a constant from 'orders'.
  • Then simulate the fix: create a 'shared' module containing TAX_RATE = 0.08.
  • Create an 'orders' module that imports TAX_RATE from shared and defines total_with_tax(amount) returning amount * (1 + TAX_RATE).
  • Print orders.total_with_tax(100), rounded to 2 decimal places.

Watch out for

  • The explanation text is not strictly checked — focus your effort on getting the simulated module structure and final number correct.
  • Do not have 'orders' and a hypothetical 'inventory' module import from each other in your simulation — that defeats the purpose of the exercise.
Hints (4)
  • For the explanation, a short printed paragraph is fine — the checker looks for the numeric result at the end, not exact wording.
  • shared.TAX_RATE is a plain module attribute — access it as shared.TAX_RATE inside orders' exec'd code, or capture it in the module dict before exec.
  • 100 * 1.08 = 108.0
  • The key lesson: extracting the shared value breaks the cycle because neither module depends on the other anymore.

Stretch — Package with __init__ Re-exports stretch

A package's __init__.py commonly re-exports selected names from submodules so users can write 'from mypackage import thing' instead of reaching into submodules directly.

# mypackage/__init__.py
# from .shapes import area_circle
# from .formatting import money
#
# usage: from mypackage import area_circle, money

What to do

  • Simulate a package 'geometry' with two submodules ('shapes' and 'formatting', reusing Part 1's definitions).
  • Simulate an __init__-style re-export: create a top-level module 'geometry' whose namespace includes area_circle, area_square, and money pulled directly from the submodules (i.e. geometry.area_circle should work without going through geometry.shapes).
  • Print geometry.money(geometry.area_circle(5)).
  • Print a sorted list of the public names available directly on 'geometry' (just the three function names).

Watch out for

  • Only include the three re-exported names in the final printed list — not any internal helpers.
  • The list must be printed with Python's default list repr, e.g. ['area_circle', 'area_square', 'money'].
Hints (4)
  • After building shapes and formatting modules, assign geometry.area_circle = shapes.area_circle etc., directly on the geometry module's __dict__ — this is exactly what a real __init__.py's import statements accomplish.
  • 3.14159 * 5 ** 2 = 78.53975, formatted to 2 decimals as 78.54.
  • sorted(['area_circle', 'area_square', 'money']) gives the alphabetical order shown.
  • This pattern is why you can 'from requests import get' instead of 'from requests.api import get'.
