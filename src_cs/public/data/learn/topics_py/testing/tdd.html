Test-Driven Development (TDD)

Test-Driven Development (TDD) is a way of writing code. First you write the test. Then you write the code. It may feel strange at first — but it is very powerful.

1. What is TDD?

TDD is a simple cycle:

  • 1. Write a test
  • 2. Run it (it fails)
  • 3. Write the code
  • 4. Run it again (it passes)
  • 5. Improve the code
Important Idea
In TDD, tests are written before the real code.

2. Why Use TDD?

TDD helps you:

  • Think clearly before coding
  • Avoid writing unnecessary code
  • Build safer programs

It forces you to ask:
“What should this function really do?”

3. Example: Building is_even()

We want to create a function called is_even(n). It should return True if a number is even, and False if it is odd.

Step 1: Write the Test First

# test_numbers.py

def test_is_even():
    assert is_even(4) == True
    assert is_even(3) == False

If you run this now, it will fail. Why? Because is_even does not exist yet.

This is normal in TDD.

4. Step 2: Write the Simplest Code

# numbers.py

def is_even(n):
    return n % 2 == 0

Now run the test again. 

pytest

If everything is correct, the test passes.

Key Rule
Write the simplest code that makes the test pass. Do not add extra features.

5. Step 3: Improve the Code (Refactor)

After the test passes, you can improve the code.

Example:

  • Better variable names
  • Cleaner structure
  • Remove repetition

But remember:

Important Rule
Never change behavior during refactoring. Tests must still pass.

6. Advantages and Challenges

Advantages:

  • Cleaner design
  • Fewer bugs
  • More confidence when changing code

Challenges:

  • Feels slower at first
  • Requires discipline
  • Hard for beginners

TDD is not magic. But when used correctly, it improves code quality.

7. Small Exercise

1. Write a test first for a function divide(a, b). 2. The function should return the result of division. 3. Write tests for:

  • Normal division
  • Division by 1
  • Division by zero (should raise an error)

4. Run the tests. 5. Then write the function. 6. Make the tests pass.

Final Thought
In normal coding, you write code and hope it works. In TDD, you prove it works.
