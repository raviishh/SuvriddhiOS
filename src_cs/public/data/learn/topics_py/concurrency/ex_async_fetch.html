Async Fetch Simulator

Simulate concurrent I/O-bound requests using asyncio, without needing real network access.

Part 1 — Sequential vs Concurrent Awaits

await pauses a coroutine without blocking the whole program. asyncio.gather() runs multiple coroutines concurrently and waits for all of them.

import asyncio

async def fetch(name, delay):
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    result = await fetch("A", 0.1)
    print(result)

asyncio.run(main())
Output:
A done

What to do

  • Write an async function fetch(name, delay) that awaits asyncio.sleep(delay) then returns f'{name} done'.
  • Write an async main() that awaits fetch('A', 0.2), fetch('B', 0.2), and fetch('C', 0.2) SEQUENTIALLY (one after another with separate await statements), timing the total with time.time().
  • Print 'Sequential total >= 0.6' if the elapsed time is at least 0.6 seconds (allow a small margin), else print the elapsed time directly.
  • Run it with asyncio.run(main()).

Watch out for

  • Do not use asyncio.gather() in this part — the point is to observe sequential (additive) timing first.
  • Timing-based checks can be slightly flaky on slow machines — the check allows some tolerance below the exact 0.6s figure.
Hints (4)
  • Sequential awaiting means: r1 = await fetch(...); r2 = await fetch(...); r3 = await fetch(...) — each one waits for the previous to finish.
  • Three 0.2-second sleeps done sequentially take roughly 0.6 seconds total.
  • Use start = time.time() before and elapsed = time.time() - start after all three awaits.
  • A small margin (e.g. elapsed >= 0.55) protects against minor timing jitter in different environments.

Part 2 — Concurrent Fetching with gather()

asyncio.gather(*coroutines) schedules all of them to run concurrently, so the total time is closer to the longest single task rather than the sum of all of them.

async def main():
    results = await asyncio.gather(fetch("A", 0.2), fetch("B", 0.2), fetch("C", 0.2))
    print(results)

What to do

  • Reuse fetch(name, delay) from Part 1.
  • Write main() that awaits asyncio.gather(fetch('A', 0.2), fetch('B', 0.2), fetch('C', 0.2)) and times the total.
  • Print the list of results.
  • Print 'Concurrent total < 0.4' if elapsed is under 0.4 seconds, else print the elapsed time.

Watch out for

  • gather() does not guarantee execution order, only result order — 'B' might actually finish before 'A' internally, but results[0] is still A's result.
  • This part must show a clear speedup versus Part 1 — if your total is still close to 0.6s, you're likely still awaiting sequentially somewhere.
Hints (4)
  • asyncio.gather() preserves the order of results matching the order of the coroutines passed in, regardless of which finishes first.
  • Because all three sleeps run concurrently, the whole gather takes about 0.2 seconds, not 0.6.
  • Print the results list exactly as returned by gather — it will be ['A done', 'B done', 'C done'].
  • 0.4 seconds gives comfortable margin above the expected ~0.2s concurrent runtime.

Stretch — Bounded Concurrency with a Semaphore stretch

Unbounded concurrency can overwhelm a real server. asyncio.Semaphore(n) limits how many coroutines run a critical section at once, simulating a connection pool or rate limit.

sem = asyncio.Semaphore(2)

async def limited_fetch(name, delay):
    async with sem:
        await asyncio.sleep(delay)
        return name

What to do

  • Create asyncio.Semaphore(2) to allow at most 2 concurrent fetches.
  • Write limited_fetch(name, delay) that acquires the semaphore, sleeps for delay, then returns name.
  • Launch 6 fetches, each with delay=0.1, using asyncio.gather() wrapped around limited_fetch for names 'A' through 'F'.
  • With max concurrency 2 and 6 tasks of 0.1s each, the total time should be roughly 0.3 seconds (3 batches of 2).
  • Print 'Bounded total >= 0.25' if elapsed is at least 0.25 seconds (proving the semaphore is actually limiting concurrency), else print the elapsed value.

Watch out for

  • Creating an asyncio.Semaphore() outside of a running event loop can raise errors on some Python versions — construct it inside your async main() function to be safe.
  • The 0.25s threshold leaves margin below the expected ~0.3s while still clearly exceeding the ~0.1s unbounded case, proving the semaphore had an effect.
Hints (4)
  • async with sem: acquires and releases the semaphore automatically, even if an exception occurs inside the block.
  • 6 tasks / 2 concurrent slots = 3 sequential batches, each taking ~0.1s, totalling ~0.3s.
  • Without the semaphore, all 6 would run concurrently and finish in ~0.1s total — the semaphore is what forces the extra time.
  • Create the Semaphore inside the same event loop as the coroutines that use it (i.e. inside main(), or ensure it's created after asyncio.run starts) to avoid loop-binding issues.
