Thread vs Process Benchmarking

Compare threading and multiprocessing on CPU-bound vs I/O-bound work, and see the GIL's effect firsthand.

Part 1 — Spawning and Joining Threads

threading.Thread(target=fn, args=(...)) creates a thread; .start() begins it, .join() waits for it to finish.

import threading

def worker(n):
    print(f"Worker {n} done")

threads = [threading.Thread(target=worker, args=(i,)) for i in range(3)]
for t in threads:
    t.start()
for t in threads:
    t.join()
Output:
Worker 0 done
Worker 1 done
Worker 2 done

What to do

  • Write a function accumulate(results, index, value) that stores value into results[index].
  • Create a shared list results = [0, 0, 0, 0].
  • Spawn 4 threads, each calling accumulate with a different index and value = index ** 2.
  • Start all threads, join them all, then print results.

Watch out for

  • Thread execution order is not guaranteed, but since each thread writes to its own index, the final result is deterministic regardless of order.
  • Do not sort or modify results after threads finish — print it in the natural index order [0,1,2,3].
Hints (4)
  • results = [0, 0, 0, 0] is a shared mutable list all threads can write into.
  • threading.Thread(target=accumulate, args=(results, i, i**2))
  • Always start() all threads first, then join() all of them — joining immediately after starting each one would serialise the work.
  • Because each thread writes to a distinct index, there's no race condition here even though the list is shared.

Part 2 — A Race Condition and Its Fix

The GIL protects individual bytecode operations, but 'x += 1' is not atomic — it's a read, a modify, and a write. Multiple threads can interleave these steps and lose updates. A Lock prevents this.

import threading
counter = 0
lock = threading.Lock()

def increment():
    global counter
    with lock:
        counter += 1

What to do

  • Write increment_unsafe() that increments a global counter (starting at 0) by 1, called from 10 threads each incrementing 1000 times, WITHOUT a lock.
  • Print the counter value after all threads join (it may or may not equal 10000 due to the race condition — that's expected and fine to report as-is).
  • Reset the counter to 0, then repeat using a threading.Lock() around the increment.
  • Print the final locked counter value: it must be exactly 10000.

Watch out for

  • The unsafe counter's final value is not deterministic — do not hard-code an expected number for it; just print whatever it computes.
  • Make sure to actually reset counter to 0 before the locked run, or your total will be inflated by leftover value from the unsafe run.
Hints (4)
  • Use 'global counter' inside each function to modify the module-level variable.
  • with lock: counter += 1 ensures only one thread modifies counter at a time.
  • 10 threads x 1000 increments = 10000 total increments expected.
  • The unsafe version's printed value is informational only — the checker looks for '10000' appearing somewhere in the output, which the locked version guarantees.

Stretch — CPU-Bound Multiprocessing Speedup stretch

For CPU-bound work, threads don't help because of the GIL — only one thread runs Python bytecode at a time. multiprocessing.Pool sidesteps this by using separate processes, each with its own GIL.

from multiprocessing import Pool

def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n**0.5) + 1):
        if n % i == 0:
            return False
    return True

with Pool(4) as p:
    results = p.map(is_prime, range(2, 20))

What to do

  • Write is_prime(n) as shown above.
  • Using multiprocessing.Pool(processes=4), compute is_prime for every number from 100000 to 100050 (inclusive range of 51 numbers).
  • Print the list of numbers in that range that are prime.
  • Print the count of primes found.

Watch out for

  • multiprocessing.Pool has real process-startup overhead — for very small workloads, threads or even sequential code can outperform it.
  • Each worker process gets a COPY of shared state — modifying a global inside a worker will not be visible in the parent process.
Hints (4)
  • with Pool(processes=4) as pool: results = pool.map(is_prime, range(100000, 100051))
  • Zip the range with the results to filter: [n for n, prime in zip(range(100000, 100051), results) if prime]
  • On some platforms (notably Windows and when run as a script), multiprocessing code must be guarded by if __name__ == '__main__': — mention this even if your environment doesn't strictly require it here.
  • The exact prime list in [100000, 100050] can be verified independently — there are 3 primes in that range.
