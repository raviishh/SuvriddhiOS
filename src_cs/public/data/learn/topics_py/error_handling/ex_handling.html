Exception Handling Drills

Wrap risky operations in try/except and handle errors gracefully.

Part 1 — Safe Division

Dividing by zero raises ZeroDivisionError. Wrap it in try/except to handle it without crashing.

try:
    result = 10 / 0
except ZeroDivisionError:
    print("Cannot divide by zero")
Output:
Cannot divide by zero

What to do

  • Write a function safe_divide(a, b) that returns a / b.
  • If b is 0, return the string "Error: division by zero".
  • Print safe_divide(10, 2) and safe_divide(5, 0).

Watch out for

  • Return the error string — don't raise, don't print from inside the function.
  • The result of 10 / 2 is 5.0, not 5 — Python 3 always returns float for /.
Hints (4)
  • Use try/except ZeroDivisionError inside the function.
  • Return the error string, don't print it inside the function.
  • 10 / 2 returns 5.0 (float division) — that is correct.
  • You could also check if b == 0 before dividing, but using try/except is more Pythonic (EAFP: Easier to Ask Forgiveness than Permission).

Part 2 — finally Always Runs

The finally block executes whether or not an exception occurred — useful for cleanup.

try:
    x = int("abc")
except ValueError:
    print("Bad value")
finally:
    print("Done")
Output:
Bad value
Done

What to do

  • Write code that tries to convert the string "123abc" to int.
  • Catch the ValueError and print: Conversion failed.
  • In a finally block, print: Attempted conversion.

Watch out for

  • Include the period at the end of both printed lines.
  • finally is not the same as else — else only runs on success, finally always runs.
Hints (4)
  • int("123abc") raises ValueError.
  • finally runs no matter what.
  • The finally block runs even when there is no exception — try passing '123' to see it succeed, then check finally still runs.
  • You can also use the else clause: try/except/else/finally — else runs only when no exception occurred.

Stretch — Custom Exception Hierarchy stretch

Custom exceptions let you communicate domain-specific errors clearly. They should inherit from Exception (or a more specific built-in) and optionally carry extra data.

class AppError(Exception):
    pass

class ValidationError(AppError):
    def __init__(self, field, message):
        self.field = field
        super().__init__(f"{field}: {message}")

What to do

  • Define a base class AppError(Exception).
  • Define ValidationError(AppError) with field and message attributes.
  • Define NotFoundError(AppError) with resource and id_ attributes.
  • Write validate_age(age) that raises ValidationError('age', 'must be positive') if age <= 0.
  • Write find_user(id_) that raises NotFoundError('user', id_) if id_ not in {1, 2, 3}.
  • Call both with invalid inputs, catch each with except AppError as e and print str(e).

Watch out for

  • Do not catch Exception directly — catching AppError is more precise.
  • str(e) returns the message passed to __init__ via super() — make sure you call super().__init__.
Hints (4)
  • super().__init__(message) passes the message to the base Exception so str(e) works.
  • For NotFoundError: super().__init__(f"{resource}: {id_}")
  • except AppError catches both ValidationError and NotFoundError because of inheritance.
  • Catch them in two separate try/except blocks to print each error independently.
