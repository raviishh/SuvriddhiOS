Project Architecture — Organizing Real Python Projects

In this lesson, you will learn how to organize your code like a real developer. Clean structure makes programs easier to read, fix, and grow.

1. The problem with messy code

Beginners often write everything in one file.

big_project.py

This works at first. But when the project grows:

  • The file becomes very long
  • It becomes hard to understand
  • It becomes hard to fix bugs
Simple truth
Messy structure leads to messy thinking.

2. Separating responsibilities

Each file should have one clear purpose.

  • One file handles the database
  • One file handles user input
  • One file runs the app

This idea is called separation of concerns.

3. A clean project structure

task_manager/
│
├── app.py
├── database.py
├── models.py
├── utils.py
├── requirements.txt
└── README.md

Explanation:

  • app.py → main program
  • database.py → database logic
  • models.py → data structures
  • utils.py → helper functions
  • requirements.txt → project dependencies

4. Splitting code into modules

Suppose this is inside database.py: 

import sqlite3

def connect():
    return sqlite3.connect("tasks.db")

def create_table():
    conn = connect()
    cursor = conn.cursor()
    cursor.execute("""
    CREATE TABLE IF NOT EXISTS tasks (
        id INTEGER PRIMARY KEY,
        description TEXT
    )
    """)
    conn.commit()
    conn.close()

Now in app.py:

from database import create_table

create_table()

print("App started!")

Files now work together cleanly.

5. Using packages (folders with __init__.py)

When projects grow larger, we use folders.

task_manager/
│
├── app.py
└── core/
    ├── __init__.py
    ├── database.py
    └── models.py

The file __init__.py tells Python that the folder is a package.

Import example:

from core.database import create_table

6. Virtual environments

Different projects may need different libraries. A virtual environment keeps them separate. 

python -m venv venv
source venv/bin/activate   # Mac/Linux
venv\Scripts\activate      # Windows

Install packages inside the environment:

pip install flask

Save dependencies:

pip freeze > requirements.txt

7. Refactoring example

Before:

big_app.py (500 lines)

After:

project/
├── app.py
├── database.py
├── services.py
└── models.py

Same functionality. Cleaner structure.

Important habit
Refactor early. Do not wait until the project becomes unmanageable.

8. Small exercises

  • Take one of your old projects and split it into at least 3 files.
  • Create a folder called core and move database code inside it.
  • Create a virtual environment for your project.
Final thought
Writing code is only half the skill. Organizing it is what makes you professional.
